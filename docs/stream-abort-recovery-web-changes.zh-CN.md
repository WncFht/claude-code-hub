# 半截断流修复后，Web 控制台可见变化说明

## 背景

这次修复主要解决两类问题：

1. 上游流式响应中途断开时，Hub 之前容易把问题记成含义不清的中断，或者误记成客户端取消。
2. 旧数据库在启动迁移阶段可能因为 `drizzle.__drizzle_migrations` 为空而重复回放早期迁移，出现 `keys already exists`、`system_settings` 缺列等启动噪音。

本次改动已经上线到运行中的 Claude Code Hub 容器，重点影响的是日志归因与后台启动稳定性。Web 界面没有新增独立页面，但原有页面里的信息会更准确。

## 你在 Web 上会看到什么

### 1. 控制台 `流量 > 日志` 页面

路径：

- `/zh-CN/console/traffic/logs`
- 旧地址 `/zh-CN/dashboard/logs` 会自动跳转到这里

这是本次改动最直接的可见位置。

#### 现在会看到的变化

- 失败请求仍然会在日志表格中显示为红色 `502`、`499` 等状态码。
- 但这些状态背后的错误含义会更准确：
  - 真实客户端取消才会记为 `CLIENT_ABORTED`
  - 流中途被上游断开会记为 `STREAM_UPSTREAM_ABORTED`
  - CCH 在流处理过程中出错会记为 `STREAM_PROCESSING_ERROR`
  - 流式空闲超时会记为 `STREAM_IDLE_TIMEOUT`
  - 首包后整体响应超时会记为 `STREAM_RESPONSE_TIMEOUT`
- 因此，日志页里“看起来像客户端自己停掉”的失败数量会减少，转而出现更真实的 `STREAM_*` 类型。

#### 建议怎么查看

在日志页里优先这样看：

- 用状态码过滤 `502`
- 观察失败行数量
- 点开失败请求的详情弹窗

在详情里，重点看两个区域：

- 摘要页里的错误信息字段
- 逻辑链路/Logic Trace 里的失败原因

这次修复后，这两个位置会更稳定地反映真实原因，而不是把很多中途中断都糊成“客户端取消”。

### 2. 控制台 `系统 > 日志` 页面

路径：

- `/zh-CN/console/system/logs`
- 旧地址 `/zh-CN/settings/logs` 会自动跳转到这里

这个页面的 UI 本身没有新增控件，但它对应的后台日志环境会更干净。

#### 现在会看到的变化

- 容器重启后，不应再反复出现迁移硬失败导致的启动异常。
- 之前常见的这类问题应该消失或显著减少：
  - `relation "keys" already exists`
  - `system_settings` 缺少 `codex_priority_billing_source`

换句话说，这一页不是“新增功能”，而是用于观察“启动环境更稳定了、异常噪音更少了”。

## 不会在 Web 上直接新增的内容

以下改动对体验有影响，但不是通过“新页面/新按钮”体现：

- Response API 流在中途失败时，Hub 会向客户端补发合成的 `response.failed` 终止事件
- 这主要体现在 VS Code / Codex / 其它 Response API 客户端里
- Web 控制台不会单独新增一个“response.failed 事件查看器”

因此，Web 侧最核心的变化仍然是“日志解释更准确”。

## 这次上线后建议你重点观察的指标

### 在 `流量 > 日志` 里看

- 最近失败请求里，`499 CLIENT_ABORTED` 是否更接近真实用户手动中断
- `502` 失败里是否开始出现明确的 `STREAM_*` 分类
- 打开失败详情时，错误信息是否比之前更容易区分“上游断流”和“客户端取消”

### 在 `系统 > 日志` 里看

- 重启后是否还出现迁移失败
- 是否还出现缺列降级读取相关报错

## 验收结论

本次可视验收的结果是：

- 登录页样式没有变化，这是预期行为
- `流量 > 日志` 页面仍是主要观察入口，失败状态行能正常展示
- `系统 > 日志` 页面可正常访问
- 部署后的容器健康检查正常，迁移已成功完成，没有再出现启动即退出的问题

如果你后续要继续验收“半截输出停住”这件事，最推荐的流程是：

1. 在真实 Codex/VS Code 客户端里复现一次流式失败
2. 回到 `流量 > 日志`
3. 过滤 `502`
4. 打开对应请求详情，确认错误已落成 `STREAM_*` 而不是泛化的客户端中断
