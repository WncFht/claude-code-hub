## Context

现有使用文档页面 `/usage-doc` 仍然存在，并且内容、布局和登录态适配都已经可用。问题不在于文档内容丢失，而在于登录后的主工作区已经迁移到 `/console`，而当前 console 共用外壳没有提供任何文档入口，导致用户在迁移后的主前端中无法再自然发现文档。

当前 console 的导航由共享外壳统一渲染，适合在共用位置一次性恢复入口。相比新建一套 console 文档 screen，把入口接到现有 `/usage-doc` 页面可以用更小的改动恢复可访问性，并避免复制文档内容。

## Goals / Non-Goals

**Goals:**

- 让管理员和允许访问 Web UI 的普通用户都能在当前 console 前端中看到文档入口
- 复用现有 `/usage-doc` 页面，恢复文档的可发现性
- 把改动限制在共享导航外壳和必要测试中，尽量避免扩大范围

**Non-Goals:**

- 重写或迁移现有文档页面内容
- 为文档建立新的 console 模块、独立数据预加载链路或新的运行时 screen
- 改造整个 console 导航体系或旧版 dashboard 文档页布局

## Decisions

### 1. 在 `ConsoleHeader` 中加入全局文档入口

文档入口将放在所有 console 页面都能看到的共享 `ConsoleHeader` 中，而不是挂在某个只对管理员可见的子页面下。这样可以同时覆盖管理员和普通 Web UI 用户，并确保用户进入任意 console 页面后都能直接找到文档。

选择理由：

- `ConsoleHeader` 天然覆盖所有 console 页面，改动集中。
- 比放在某个模块内部更符合“恢复可发现性”的目标。
- 不需要为文档新增模块级信息架构。

备选方案：

- 新增一个 console 模块或子路由：放弃，因为这会引入新的运行时 screen、预加载和导航结构，超出本次最小修复范围。
- 只在旧版 dashboard header 中保留入口：放弃，因为登录后的主入口已经不再依赖旧 header。

### 2. 入口直接指向现有 `/usage-doc` 页面

文档入口将直接导航到现有 `/usage-doc` 页面，而不是在 console 内重新包一层文档 screen。

选择理由：

- 现有文档页面已经可用，且内容完整。
- 避免复制文档组件、目录逻辑和登录态处理。
- 能用最小改动恢复访问路径。

备选方案：

- 在 console stage 中嵌入或重建文档页面：放弃，因为会带来布局复用、目录滚动和内容同步问题，复杂度明显更高。

### 3. 文案复用现有 i18n 体系，并为入口补测试

入口标签将复用现有 i18n 文案键，而不是硬编码文本。测试将覆盖入口的存在性和目标路径，避免后续 console 外壳调整时再次把文档入口移除。

选择理由：

- 符合仓库的 i18n 规则。
- 共享外壳改动最容易在重构中被遗漏，测试能提供回归保护。

## Risks / Trade-offs

- [文档页面仍然位于 console 外部路由] -> 这是有意的最小改动，先恢复访问能力；如果后续需要深度整合，再单独开 change 迁移文档体验。
- [入口放置位置不够显眼] -> 选择共享外壳中的常驻区域，并通过测试固定其存在，优先解决“找不到”问题。
- [未来文档路径调整导致入口失效] -> 将目标路径集中在共享实现中，并补路由断言测试。

## Migration Plan

1. 在 `ConsoleHeader` 中加入文档入口，并接到现有 `/usage-doc`。
2. 复用现有 i18n 文案或补齐所需文案键。
3. 为共享外壳补测试，验证登录后的 console 中存在文档入口且目标路径正确。
4. 手动验证管理员和普通 Web UI 用户都能从当前前端到达文档页面。
